Protocols in Practice: TLS, SSH, VPN, PKI, Email, and Code Signing

Concrete protocol-level examples showing where PQC changes appear and what breaks if the surrounding ecosystem is ignored.

  • Describe how PQC appears in common protocols.
  • Identify common interoperability and certificate-chain risks.
  • Separate protocol migration from application feature work.

Educational reference only. Production cryptography decisions need current standards checks, vendor validation, security review, interoperability testing, and rollback planning.

1

TLS and HTTPS

For HTTPS, PQC usually appears first in key establishment experiments and hybrid modes. Later, certificate signatures and certificate-chain support become a major part of the transition.

The migration surface includes browsers, mobile OS TLS stacks, CDNs, load balancers, WAFs, proxies, application servers, observability tools, certificate automation, and incident response playbooks.

A real test should measure handshake size, latency, failure rate, client compatibility, middlebox behavior, downgrade resistance, and certificate parsing.

ClientBrowser, app, device.
MiddleboxesCDN, proxy, WAF.
OriginServer TLS stack.
TelemetryFailures, latency, downgrade.

Concrete example

An API may support hybrid TLS at the origin but fail when traffic passes through an older enterprise proxy.

Common pitfall

Do not declare success based only on a direct curl test from a developer laptop.

Takeaway

TLS migration is path-dependent.

2

SSH and VPN

SSH and VPNs often protect administrative or internal access, so they can be high priority even if not public websites. Their key exchange, host keys, certificates, and client compatibility all matter.

Many environments have old clients, appliances, jump hosts, scripts, and automation that assume specific algorithms. PQC migration must be staged to avoid locking out administrators or silently falling back to weak modes.

Policy needs to define which algorithms are preferred, which are allowed only temporarily, and which are blocked.

User/clientAdmin tools and scripts.
GatewayVPN or jump host.
Internal targetProtected systems.
PolicyAllowed algorithms and fallback.

Concrete example

A jump host can support a hybrid key exchange, but an old automation client may fail unless the rollout first upgrades client tooling.

Common pitfall

Do not remove legacy support before confirming emergency access paths.

Takeaway

Administrative access protocols require careful compatibility and rollback planning.

3

Email and messaging trust

Email security uses several cryptographic layers, including TLS transport, DKIM signatures, S/MIME or PGP in some environments, and identity or domain authentication. Each layer has different migration timing.

Messaging systems may use end-to-end encryption, identity keys, device keys, prekeys, signatures, and server transport protection. PQC migration depends heavily on protocol design and client support.

The learner should identify which layer protects confidentiality, which proves identity, and which creates forward secrecy.

TransportTLS between servers.
DomainDKIM and SPF/DMARC ecosystem.
End-to-endS/MIME, PGP, messaging protocols.
ClientKey and certificate support.

Concrete example

A mail server may adopt hybrid TLS before S/MIME certificate ecosystems are ready for PQC signatures.

Common pitfall

Do not assume transport encryption solves end-to-end message confidentiality.

Takeaway

Email and messaging have layered cryptography with different PQC paths.

4

Code signing and package ecosystems

Code signing protects software supply chains. A signature may be verified by package managers, operating systems, bootloaders, CI/CD tools, app stores, browsers, security scanners, and customer environments.

PQC signature migration must account for artifact formats, signature size, timestamping, transparency logs, offline verification, HSM support, and old verifiers. It is often a long-tail migration.

A safe transition can use parallel signatures, metadata versioning, staged verifier updates, and explicit policy.

BuildCreates artifact.
SignPrivate key in HSM/KMS.
DistributeRegistry, app store, update server.
VerifyOS, device, customer, scanner.

Concrete example

A container signing system may support a new signature algorithm while a customer's policy engine still rejects the artifact because it cannot parse the new signature metadata.

Common pitfall

Do not update the signing pipeline before all critical verifiers are ready.

Takeaway

Code signing is a supply-chain ecosystem, not a single tool.
