1

Algorithms are public; keys are protected

Modern cryptography does not rely on hiding the algorithm. Public standards are reviewed by many experts, implemented by many vendors, and tested across many environments. Security depends on strong design plus secret or private key material that remains protected.

A key is not just a string. It has origin, owner, allowed purpose, creation time, rotation rules, backup method, revocation path, and destruction requirements. Key management failures often cause more damage than algorithm weakness.

PQC does not change this principle. ML-KEM, ML-DSA, and SLH-DSA are public. The private keys, seeds, randomness, and implementation behavior must still be protected.

Public standardReviewed design.
Secret/private keyProtected material.
Allowed purposeEncrypt, sign, verify, or wrap.
LifecycleGenerate, store, rotate, revoke.

Concrete example

Publishing the AES algorithm does not let attackers decrypt AES-256. Accidentally logging the AES key may.

Common pitfall

Do not invent a secret algorithm because it feels safer. Use reviewed standards and protect keys correctly.

Takeaway

The algorithm is the public recipe; the key is the protected control.

2

Hashes, MACs, and password hashing

A cryptographic hash creates a fixed-size digest. It is useful for fingerprints, integrity checks, Merkle trees, transparency logs, and signature internals. It is not reversible encryption.

A message authentication code, or MAC, proves integrity using a shared secret. A hash alone can show data changed if the expected digest is trusted; a MAC proves the checker knows the shared secret.

Password hashing is a separate use case. Passwords should use password-hashing schemes with salts and work factors, not plain fast hashes. This matters because the same word 'hash' appears in several different contexts.

HashFingerprint.
MACShared-secret integrity.
Password hashSlow and salted.
SignaturePublic verification of origin.

Concrete example

A download page may publish a SHA-256 digest. An API request may use an HMAC to prove a shared secret. A login system should use a password-hashing design, not just SHA-256(password).

Common pitfall

Do not confuse hash integrity with authentication. If an attacker can replace both a file and its posted digest, the digest does not prove origin.

Takeaway

Hashes fingerprint; MACs authenticate with a shared secret; password hashing protects low-entropy secrets.

3

Digital signatures and verification policy

A digital signature is created with a private signing key and checked with a public verification key. Anyone can verify, but only the private key holder should be able to sign.

Verification policy is as important as signature creation. A verifier must know which algorithms are accepted, which keys are trusted, which uses are allowed, whether the signature is expired or revoked, and whether legacy fallback is permitted.

PQC signatures matter because existing RSA, ECDSA, and EdDSA signatures are in certificate chains, firmware, documents, code releases, package repositories, and identity systems.

SignerPrivate key creates signature.
ArtifactFirmware, cert, package, document.
VerifierPublic key checks it.
PolicyAllowed algorithms and keys.

Concrete example

A device bootloader may verify firmware before installation. If the bootloader cannot understand future PQC signatures, the device may need an agility update before a PQC signing policy can be enforced.

Common pitfall

Do not only ask 'can we sign?' Ask 'can every verifier parse and enforce the new signature policy?'

Takeaway

Signature migration is often constrained by old verifiers.

4

TLS as a bundle of cryptographic jobs

TLS is the protocol behind HTTPS and many APIs. It combines identity, negotiation, key establishment, encryption, integrity, and downgrade protection. That is why changing TLS for PQC is more than swapping a function call.

A simplified handshake negotiates supported parameters, authenticates the server through a certificate, establishes a shared secret, derives symmetric keys, and then protects records. PQC can affect the key establishment and certificate-signature parts.

Hybrid key exchange is a common transition model because it combines classical and PQC contributions. It needs protocol-defined combination and downgrade protection.

NegotiatePick supported options.
AuthenticateCertificate proves identity.
Establish secretClassical, PQC, or hybrid.
Protect recordsSymmetric encryption.

Concrete example

A hybrid TLS pilot may work directly between client and server but fail through a middlebox that rejects larger handshakes or unknown extensions.

Common pitfall

Do not test only the crypto library. Test browser, mobile app, proxy, WAF, CDN, load balancer, logging, monitoring, and rollback.

Takeaway

Protocol migration must be tested on the real path.
