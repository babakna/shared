Cryptographic Building Blocks Deep Dive

A tour of the parts every secure system is assembled from — and, crucially, which parts a post-quantum migration touches. We finish by walking through a real TLS handshake step by step.

  • Tell keys, hashes, MACs, signatures, and certificates apart and know what each is for.
  • See where implementation and operational mistakes defeat strong algorithms.
  • Identify exactly which handshake steps PQC changes.

Try it: walk through a TLS 1.3 handshake

This is what happens in a fraction of a second before HTTPS shows a page. Notice which steps a quantum computer threatens (marked PQC).

Read the pattern

Of five steps, PQC changes the key exchange and the signatures (steps 1–4). The bulk data encryption (step 5) mainly needs adequate key size. This is the shape of nearly every protocol migration.

1

Algorithms are public; keys are protected

Modern cryptography does not hide the algorithm. It hides the key.

Public, reviewed standards get analyzed by thousands of experts and implemented by many vendors. Security comes from a strong design plus secret key material that stays protected. A key is not just a random string — it has an origin, an owner, an allowed purpose, a creation time, rotation rules, a backup method, a revocation path, and destruction requirements. Key-management failures cause more real breaches than algorithm weaknesses.

Concrete example

Publishing the AES specification does not help anyone decrypt AES-256. Accidentally writing the AES key into a log file does. The recipe is public; the key is the control.

Common pitfall

Inventing a secret in-house algorithm because secrecy "feels safer." Unreviewed crypto fails in ways experts would have caught. Use vetted standards and protect the keys.

This idea is old enough to have a name: Kerckhoffs's principle (1883) says a system should stay secure even if everything about it except the key is public. It is why NIST runs multi-year, worldwide public competitions to pick algorithms — ML-KEM and ML-DSA were chosen exactly this way, after years of cryptographers trying to break every candidate in the open. "Security through obscurity" fails because you cannot patch a secret design once it leaks, and you never get the free global peer review.

Because the key carries all the security, it needs a full lifecycle: generated from strong randomness, stored so it cannot be exported (ideally in an HSM), used only for its declared purpose, rotated on a schedule, revocable if compromised, and destroyed at end of life. A brilliant algorithm with a key sitting in a world-readable config file is worthless. This is why PQC-301 spends so much time on key handling rather than the math.

PQC changes nothing about this principle. ML-KEM, ML-DSA, and SLH-DSA are fully public. Their private keys, seeds, and randomness still need protection — and PQC adds new concerns like larger key material and side-channel-safe implementations (covered in PQC-301).

Why "public algorithm" is a strength, not a weakness

Newcomers often feel that publishing exactly how encryption works must make it easier to break. The opposite is true. Because the design is public, thousands of the world's best cryptographers have spent years trying to break it — and every weakness they found got fixed before you ever relied on it. A secret in-house algorithm gets reviewed by, at best, the handful of people who built it. You are choosing between a lock tested by the entire world and a lock tested by five colleagues on a deadline. The public one wins every time.

2

Hashes, MACs, and password hashing

Three different jobs that all get sloppily called "hashing."

Hash

A one-way fingerprint of data (e.g. SHA-256). Same input → same digest. Used for integrity checks, transparency logs, and inside signatures. Not reversible and not a secret.

MAC

Proves integrity and that the sender knew a shared secret (e.g. HMAC). A plain hash only detects change if you already trust the expected value; a MAC proves who computed it.

Password hashing

Deliberately slow, salted schemes (e.g. Argon2, bcrypt, scrypt) for low-entropy secrets. Never store SHA-256(password) — it is far too fast to brute-force.

Concrete example

A download page publishes a SHA-256 digest (integrity, if you trust the page). An API request carries an HMAC (proves shared-secret origin). A login system stores an Argon2 hash (protects weak passwords). Same word, three different mechanisms.

Everyday analogy

Picture three security jobs at a building. A hash is a tamper-evident seal on a package — anyone can see if it was opened, but a seal alone does not prove who sealed it. A MAC is a seal stamped with a signet ring only you and the recipient own — now it also proves it came from someone holding the shared secret. Password hashing is a deliberately stiff, slow lock on the front door: you want it to take effort to open, so a thief who steals the lock cannot try millions of keys per second. Calling all three "hashing" is like calling a seal, a signet, and a deadbolt all "the door thing."

The distinction matters in practice because using the wrong tool quietly removes the protection you thought you had. A plain hash where you needed a MAC lets an attacker who can alter the data simply recompute the hash to match — the check passes and you learn nothing. A fast hash where you needed a slow password hash means a leaked database of "hashed" passwords can be cracked in hours. No algorithm is broken in these stories; the wrong job was assigned to the right tool.

Quantum note

Hashes and MACs are symmetric-world tools: Grover pushes toward longer outputs, but they are not "broken" the way RSA is. Password hashing is unaffected by the PQC transition itself.

3

Digital signatures and verification policy

Signing is easy to talk about; the hard part is every place that must verify.

A signature is made with a private key and checked with the matching public key. Anyone can verify; only the key holder can sign. But the verifier's policy is where migrations live or die: which algorithms are accepted, which keys are trusted, which uses are allowed, whether a signature is expired or revoked, and whether legacy fallback is permitted.

SignerPrivate key produces the signature.
→
ArtifactFirmware, cert, package, or document.
→
VerifierPublic key + policy checks it.
→
Every verifierAll of them must accept the new algorithm.
Concrete example

A device bootloader verifies firmware before installing it. If the bootloader cannot parse a future PQC signature, you cannot enforce a PQC signing policy until the bootloader itself gains "agility" support — possibly a hardware refresh.

There is also a subtle detail worth knowing: you almost never sign a whole file directly. Instead you hash-then-sign — compute a fixed-size digest of the artifact, then sign the digest. That is why hashes (lesson 2) live inside signatures, and why a weak hash quietly undermines an otherwise strong signature. In the PQC transition both pieces get attention: the signature algorithm is replaced (ML-DSA/SLH-DSA), and the hash may be lengthened to keep its collision margin against Grover.

Signatures vs key exchange have different clocks

Harvest-now-decrypt-later makes key exchange urgent, because captured traffic can be decrypted later. Signatures are different: a signature only needs to be unforgeable at verification time. But long-lived signed artifacts — firmware in a 10-year device, documents that must stay provable for decades — still need PQC signatures early, because the verifier will outlive the algorithm.

Common pitfall

Asking only "can we sign with PQC?" The real question is "can every verifier in the world that reads our artifacts parse and enforce it?"

4

TLS as a bundle of cryptographic jobs

TLS is not one algorithm; it is identity + negotiation + key establishment + encryption + integrity + downgrade protection, working together.

To see why one small connection is such a big migration, list the distinct cryptographic jobs a single TLS 1.3 session performs — and note which the quantum threat touches:

  • Identity / authentication — the certificate proves the server is who it claims. PQC target (signatures).
  • Negotiation — both sides agree on versions, ciphers, and groups, with protection against a downgrade attack. Structural, not crypto-broken.
  • Key establishment — derive a shared secret without sending it. PQC target (ML-KEM).
  • Handshake signature — prove the certificate holder is live now. PQC target (signatures).
  • Bulk encryption + integrity — AEAD protects the actual data. Kept/resized.

That is why "making TLS post-quantum" is more than swapping a function call. You saw the five steps above. Hybrid key exchange is the common transition model: run classical ECDHE and ML-KEM together, combine both secrets, and you are protected if either survives. It must be protocol-defined, with downgrade protection.

Go deeper: why hybrid handshakes sometimes fail on the real network path

A hybrid ClientHello is larger because it carries an extra, bigger key-share. That can push the message past sizes that some middleboxes, load balancers, or old TLS stacks handle gracefully — a class of bugs sometimes called "TLS ossification." Symptoms: the handshake works browser-to-server in a lab but fails through a corporate proxy, a legacy VPN concentrator, or an ancient mobile OS.

Lesson: test on the real path (client, OS TLS stack, proxy, WAF, CDN, load balancer, logging, monitoring, rollback), not just the crypto library.

Common pitfall

Testing only the crypto library. The library is rarely where PQC breaks — the surrounding ecosystem is.

5

How your data actually gets encrypted (AEAD)

Once the handshake agrees on a key, one workhorse does the real protection: authenticated encryption.

The key exchange gets all the quantum attention, but it only produces a shared secret. The actual bytes of your session — every request, password, and page — are protected by AEAD: Authenticated Encryption with Associated Data. Modern AEAD ciphers like AES-256-GCM and ChaCha20-Poly1305 do confidentiality and integrity in a single pass, which is why they dominate TLS, SSH, and VPNs.

An AEAD cipher takes four inputs and produces two outputs. It needs the key (from the handshake), a unique nonce (a number used once), the plaintext, and optional associated data (headers you want authenticated but not encrypted). It outputs the ciphertext plus an authentication tag — the fingerprint that makes tampering detectable.

The one rule you must never break

Never reuse a nonce with the same key. Nonce reuse in AES-GCM can leak plaintext and even the authentication key. This is the most common catastrophic misuse of AEAD — and it has nothing to do with quantum; it is a classic implementation trap that PQC-301 revisits.

Quantum note

AEAD is symmetric. AES-256-GCM keeps a strong margin against Grover, so in the PQC transition it is resized/kept, not replaced. The quantum action is entirely in how the key was established.

6

Hybrid key establishment, step by step

The safest way to adopt a young algorithm is to not bet everything on it yet.

ML-KEM is well-analyzed but far newer than ECDH. Rather than trust it alone, most deployments run both a classical and a post-quantum key exchange in the same handshake and combine the two shared secrets with a key-derivation function (KDF). The result is secure as long as either half remains unbroken — classical protects you if an unknown flaw is found in ML-KEM, and ML-KEM protects you against a quantum computer. You only lose if both break, which is highly unlikely.

You can spot it in the wild

In a modern browser's connection details you may see a group named X25519MLKEM768. The X25519 half is classical ECDH; the MLKEM768 half is post-quantum. Both ran, and their secrets were combined — exactly the picture above.

Hands-on labs

Decide first, then reveal.

Lab 1 · Package repository

A repo posts each package's SHA-256 hash next to the download.

Why isn't publishing the hash enough to trust the package?

Recommended answer A hash only proves integrity if you already trust the expected value. An attacker who can replace the package can also replace the posted hash. You need a signature to prove maintainer origin.

Lab 2 · Legacy mobile app

A hybrid-PQC TLS pilot works in the browser but a five-year-old mobile app fails to connect.

Name likely causes.

Recommended answer The app's bundled TLS stack, the OS, a proxy on the path, or certificate pinning may not understand the new group identifiers or the larger handshake size. It is an ecosystem issue, not a server bug.

Lab 3 · CI/CD signing key

Your build pipeline signs releases that customers verify.

What must change for a PQC signing migration?

Recommended answer The signing system, its HSM/KMS, the build tools, and every consumer that verifies releases must support the new signature policy — and you need a rollback path if a consumer cannot.

Lab 4 · Why hybrid, not pure PQC?

A stakeholder asks why you are deploying X25519+ML-KEM-768 instead of ML-KEM alone, "which is the whole point."

Give the security argument for hybrid.

Recommended answer ML-KEM is well-studied but far younger than ECDH. Hybrid combines both secrets so the connection stays safe if either holds — classical covers an unknown flaw in ML-KEM, ML-KEM covers the quantum threat. You only lose if both break, which is extremely unlikely. It is defense in depth during a transition.

Lab 5 · Nonce reuse

An engineer proposes using a fixed nonce for AES-GCM "to save a few bytes per message."

Why is this dangerous, and is it a quantum issue?

Recommended answer Reusing a nonce with the same key in AES-GCM can leak plaintext relationships and even the authentication key, destroying both confidentiality and integrity. It is a classic (non-quantum) implementation catastrophe — a reminder that most real breaches come from misuse, not algorithm weakness.
