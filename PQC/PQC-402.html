What a CBOM is

A CBOM is a software bill of materials focused on cryptography.

Where an SBOM lists software components, a CBOM lists cryptographic assets and uses: algorithms, keys, certificates, protocols, libraries, hardware, and trust relationships. It captures enough to make risk decisions, and it is maintained continuously because apps, certificates, vendors, and libraries change.

SystemWhere crypto is used.
→
JobEncrypt, sign, exchange, hash.
→
DependencyLibrary, product, HSM, vendor.
→
RiskLifetime, exposure, readiness.
Takeaway

A CBOM is the evidence base for every PQC decision. No inventory, no program.

2

Minimum fields that make the inventory useful

A row without an owner or a data lifetime cannot be prioritized or actioned.

Record ownership and enough precision to act. Algorithm names must be exact — "TLS" is not enough; record the key-exchange, the certificate signature, the protocol version, the library, and any pinned or legacy clients. Also record confidence: some data comes from automated scans, some from vendor statements, some from manual review. Confidence prevents false certainty.

Concrete example

Instead of "external API uses SSL," write: "external API, TLS 1.3, X25519 key exchange, ECDSA P-256 certificate, nginx/OpenSSL 3.x, browser + mobile clients (one pinned), 1-year data, owner Team Web, status: plan hybrid." That row can be prioritized and assigned.

Common pitfall

Hiding unknowns. An "algorithm: unknown" row is not a gap to paper over — it is a work item with an owner and a discovery method.

3

Discovery methods

No single method finds all your crypto. Combine several.

MethodFindsMisses
Network/TLS scanningPublic endpoints, certificate detailsOffline signing, internal-only, embedded
Source & dependency analysisLibraries, hard-coded parametersVendor appliances, runtime config
Config/certificate inventoryCert chains, key stores, cipher configUndocumented fallbacks
Vendor questionnairesProduct support statementsActual deployed configuration
Architecture interviewsHidden workflows, key custodyScale/coverage without tooling
Concrete example

A scanner finds all your public TLS endpoints, but only an architecture interview reveals the offline root key used to sign firmware — often your highest-risk, longest-lived asset.

4

From CBOM to work queue

An inventory that does not produce actions is just documentation.

Each row should generate work: test hybrid TLS, enlarge a schema field, request a vendor roadmap, add certificate-chain monitoring, replace a weak algorithm, file an exception, or schedule device replacement. Each action carries an owner, target date, dependency, evidence requirement, and rollback plan. The dashboard then shows coverage, risk tiers, blockers, tests done, exceptions, and next-wave candidates — which is exactly the evidence PQC-401 asks for.

Takeaway

Inventory without action is not a migration program. The CBOM feeds the work queue; the work queue feeds the dashboard.

Hands-on labs

Use the builder above, then check your reasoning.

Lab 1 · Write a real row

Add a CBOM row (in the tool above) for an internet-facing payment API.

Which fields are the minimum to make it actionable?

Recommended answer System/endpoint, crypto job (key exchange + cert signature), exact algorithms and versions, library/product, client population (incl. pinned/legacy), data lifetime, exposure, owner, and PQC status/next action. Anything less cannot be prioritized or assigned.

Lab 2 · Find hidden crypto

Your automated scans report "coverage complete."

Where might crypto still be hiding?

Recommended answer Offline signing systems, embedded devices, HSM/KMS workflows, vendor appliances, custom key-wrapping, and backup/archive pipelines. Add architecture interviews to catch what scanners cannot see.

Lab 3 · Unknown to action

A row says "algorithm: unknown."

What is the correct next step?

Recommended answer Assign an owner and a discovery method with a due date. "Unknown" is a tracked work item, never an acceptable final state.
