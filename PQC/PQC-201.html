The standards baseline

NIST finalized FIPS 203, FIPS 204, and FIPS 205 as the first main post-quantum cryptography standards. FIPS 203 specifies ML-KEM, FIPS 204 specifies ML-DSA, and FIPS 205 specifies SLH-DSA.

The standards divide by job. ML-KEM is for key establishment. ML-DSA and SLH-DSA are for digital signatures. A standards-aligned plan starts by identifying the job, then selecting the relevant algorithm family and parameter set.

Additional algorithms remain important. HQC was selected by NIST as a backup KEM candidate for standardization, and Falcon/FN-DSA remains part of the signature standards watch. Production commitments should track current final status and validation support.

FIPS 203ML-KEM key establishment.
FIPS 204ML-DSA signatures.
FIPS 205SLH-DSA signatures.
WatchHQC and FN-DSA/Falcon status.

Concrete example

A VPN needing shared session secrets looks toward ML-KEM or hybrid KEM support. A firmware-signing workflow looks toward ML-DSA or SLH-DSA support.

Common pitfall

Do not say 'we use PQC' without saying which algorithm, parameter set, job, protocol, and implementation.

Takeaway

The correct algorithm depends on the cryptographic job.

KEMs: how ML-KEM is used

A key encapsulation mechanism lets a sender use a recipient public key to create a ciphertext and a shared secret. The recipient uses the private key to decapsulate and recover the same shared secret. The shared secret then feeds symmetric cryptography.

ML-KEM is derived from CRYSTALS-Kyber and uses module-lattice assumptions. Its parameter sets trade performance, size, and security strength. Implementations must preserve correct decapsulation behavior and avoid side-channel leakage.

A KEM is not the same thing as encrypting a large file. The usual pattern is KEM for key establishment, then authenticated symmetric encryption for data.

Public keyRecipient publishes.
EncapsulateSender creates secret and ciphertext.
DecapsulateRecipient recovers secret.
Derive keysUse secret for symmetric protection.

Concrete example

A server can publish an ML-KEM public key. A client encapsulates a shared secret to that key, then both sides derive keys for authenticated encryption.

Common pitfall

Do not store KEM ciphertexts or public keys in old fixed-size fields without checking actual sizes.

Takeaway

ML-KEM creates shared secrets; symmetric algorithms protect bulk data.

3

Signatures: ML-DSA and SLH-DSA

ML-DSA is a lattice-based general-purpose digital signature standard derived from CRYSTALS-Dilithium. It is expected to be a practical primary choice for many signature workflows as libraries and products support it.

SLH-DSA is a stateless hash-based signature standard derived from SPHINCS+. It has a different security foundation, which makes it valuable for diversity, but it can have larger signatures and different performance tradeoffs.

Signature choice depends on verifier constraints, artifact lifetime, signature size, verification speed, validation requirements, and ecosystem support.

ML-DSAPractical lattice signature.
SLH-DSAHash-based diversity.
VerifierMust parse and enforce policy.
Artifact lifeLong-lived signatures need long-lived verification.

Concrete example

A high-volume package repository may prefer ML-DSA when ecosystem support is mature. A conservative root-signing use case may evaluate SLH-DSA because its foundation differs from lattice assumptions.

Common pitfall

Do not choose signatures by signer convenience alone. Verification is where many long-term compatibility problems appear.

Takeaway

Plan signatures around every verifier that must accept them.

4

Hybrid and algorithm agility

Hybrid transition combines classical and PQC mechanisms so organizations can gain PQC protection while classical systems and interoperability remain available. Hybrid is most visible in key exchange, but signature transition may also involve parallel chains or multi-signature strategies.

Hybrid designs must be protocol-defined or cryptographically reviewed. The two mechanisms need safe combination, transcript binding, downgrade protection, error handling, and telemetry.

Algorithm agility means the system can change algorithms, parameter sets, identifiers, providers, and policy without rewriting everything. Agility is what makes future HQC or FN-DSA adoption practical if and when needed.

ClassicalCurrent compatibility.
PQCQuantum-resistant component.
Combine safelyProtocol-defined.
MonitorDetect failure and downgrade.

Concrete example

A company pilots hybrid TLS for internal APIs, measures handshake failure rates, then expands only after load balancers and clients prove compatible.

Common pitfall

Do not invent a custom hybrid combiner in application code without qualified cryptographic review.

Takeaway

Hybrid reduces transition risk only when protocol rigor is preserved.
