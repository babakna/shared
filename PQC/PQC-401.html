Enterprise Migration Playbook

A practical operating model for crypto inventory, prioritization, vendor management, governance, exceptions, rollout waves, monitoring, and evidence.

1

Inventory is the first deliverable

A PQC program starts with a cryptographic inventory: where cryptography is used, what algorithms and key sizes are present, which protocol or product supplies them, who owns the system, and what data or trust relationship is protected.

The inventory should include external endpoints, internal APIs, VPNs, SSH, certificates, code signing, firmware signing, identity, databases, backups, message queues, HSMs, KMS, TPMs, secure elements, vendor products, and managed services.

The first version will be incomplete. That is acceptable if it is honest about confidence and coverage. The point is to replace vague anxiety with accountable evidence.

DiscoverFind crypto use.
ClassifyJob and risk.
OwnAssign team.
EvidenceScan or confirmation.

Concrete example

An inventory row for a public API includes owner, URL, TLS stack, certificate chain, key exchange, signature algorithm, client types, data lifetime, and test status.

Common pitfall

Do not wait for perfect discovery before prioritizing obvious high-risk systems.

Takeaway

A partial inventory with owners beats a perfect plan that never starts.

2

Prioritization model

Prioritize by confidentiality lifetime, exposure, business impact, public-key dependency, migration complexity, and vendor readiness. This prevents both underreaction and chaotic overreaction.

High-priority systems often include long-lived confidential data, public or partner-exposed key exchanges, signing roots, firmware signing, identity systems, and products with slow vendor roadmaps.

Low-priority does not mean forgotten. It means tracked, assigned, and scheduled around normal refresh unless risk or support changes.

LifetimeHow long secrecy matters.
ExposureCan attackers collect traffic?
ImpactBusiness or safety consequence.
FeasibilityCan it be changed now?

Concrete example

A short-lived public content site may rank below an internal legal archive if the archive contains confidential data that must remain private for decades.

Common pitfall

Do not prioritize only by executive visibility.

Takeaway

Risk ranking should be explainable and repeatable.

3

Vendors, procurement, and contracts

Enterprises depend on vendors for operating systems, browsers, load balancers, VPNs, HSMs, cloud services, identity platforms, network equipment, mobile apps, and embedded devices. Vendor readiness can control migration timing.

Procurement questions should be specific: which algorithms, parameter sets, protocols, hybrid modes, certificate formats, validation status, telemetry, roadmap date, rollback controls, and known limitations are supported?

Contracts and architecture standards should require crypto agility. A product that cannot change algorithms without replacement becomes a future risk.

AskSpecific capability.
VerifyTest in environment.
TrackRoadmap and limits.
ContractAgility requirements.

Concrete example

A vendor saying 'quantum-safe ready' is not sufficient. Ask for release version, supported groups, certificate support, test method, and customer configuration steps.

Common pitfall

Do not accept marketing language as migration evidence.

Takeaway

Vendor claims must become testable requirements.

4

Governance, exceptions, rollout, and monitoring

Governance defines approved algorithms, deprecated algorithms, exception criteria, procurement language, evidence requirements, and review cadence. It keeps different teams from making incompatible decisions.

Exceptions are normal but must be explicit. Each exception needs a reason, risk, owner, compensating control, expiration date, and remediation plan.

Rollout should occur in waves with telemetry and rollback. Monitoring should capture handshake failures, certificate errors, unknown algorithms, signature failures, downgrade attempts, and client compatibility.

PolicyAllowed and blocked.
ExceptionTime-limited risk.
WavePilot then scale.
MonitorDetect breakage and downgrade.

Concrete example

A legacy VPN exception might expire in six months, require restricted access, list impacted users, and reference a vendor upgrade date.

Common pitfall

Do not allow permanent undocumented exceptions. They become hidden architecture.

Takeaway

Governance makes PQC migration operationally accountable.

Use-Case Labs

These are practical exercises. Read the challenge, decide your answer, then reveal the recommended answer and compare.

Executive roadmap

Challenge: Turn a request for 'PQC by next quarter' into a realistic plan.

Recommended answer: Inventory first, prioritize high-risk systems, launch pilots, document vendor blockers, and set evidence-based waves.

Vendor questionnaire

Challenge: Write three useful questions.

Recommended answer: Ask exact algorithms/parameters, protocol modes/cert support, and test/validation/rollback evidence.

Exception review

Challenge: Decide whether an exception is acceptable.

Recommended answer: It is acceptable only if owner, risk, compensating control, expiration, and remediation are documented.
