1

Telecom PQC is an ecosystem problem

Telecom systems use cryptography for subscriber identity, device identity, network function authentication, secure transport, roaming, provisioning, management APIs, cloud infrastructure, and firmware updates.

A single operator may depend on device makers, SIM/eSIM platforms, network vendors, cloud providers, roaming partners, managed service providers, certificate authorities, and regulators. PQC migration must account for these dependencies.

The right mental model is not one network box. It is a chain of trust relationships that span devices, access network, core network, edge/cloud, operations, and supply chain.

DeviceUE, IoT, gateway.
Access/coreNetwork services.
Cloud/edgeWorkloads and KMS.
PartnersRoaming and vendors.

Concrete example

A private 5G system may include IoT modules, a SIM/eSIM provisioning path, cloud-hosted core functions, vendor-managed gateways, and firmware update signing.

Common pitfall

Do not claim telecom readiness after upgrading only one TLS endpoint.

Takeaway

Network PQC readiness must be tracked across trust relationships.

2

5G and future 6G trust layers

5G service-based architecture uses APIs, certificates, network function authentication, cloud-native infrastructure, and transport security. Future 6G concepts may increase distributed software, AI-assisted operations, sensing, edge, and identity needs.

PQC does not directly change radio wave behavior. It changes the cryptographic trust mechanisms around network functions, management, device identity, provisioning, secure channels, and long-lived credentials.

A useful analysis separates access, transport, core, edge/cloud, management, and roaming. Each layer may have different owners, standards, vendors, and update schedules.

AccessRadio and transport.
CoreNetwork functions and APIs.
EdgeCloud workloads.
ManagementOperations and automation.

Concrete example

A network function calling another network function over a service-based interface may depend on TLS certificates and service identity policies different from SIM profile provisioning.

Common pitfall

Do not assume one standards update automatically updates deployed equipment and operational tools.

Takeaway

Analyze 5G/6G crypto layer by layer.

3

SIM and eSIM lifecycle

SIM and eSIM ecosystems involve secure elements, profile creation, remote provisioning, activation, remote management, backend platforms, certificates, and long device lifetimes.

PQC migration must respect constraints: message size, memory, CPU, power, certification, interoperability, and whether fielded devices can update. Some devices may remain deployed long after server-side systems are ready.

The learner should ask which trust relationships appear during profile download, activation, remote management, revocation, and backend communication.

ProfileSubscriber credentials.
Secure elementProtected environment.
ProvisioningDownload and activation.
ManagementUpdate, revoke, audit.

Concrete example

An eSIM profile download flow may involve the device, secure element, provisioning platform, operator backend, and certificates checked at multiple points.

Common pitfall

Do not assume secure elements can support larger PQC artifacts without testing.

Takeaway

SIM/eSIM PQC depends on constrained lifecycle engineering.

4

IoT, firmware, and long-lived devices

IoT fleets often have constrained hardware, long lifetimes, limited update windows, and uneven connectivity. Firmware signing and device enrollment are major PQC concerns because trust decisions may persist for years.

A migration plan should identify which models can verify new signatures, which bootloaders are updateable, which devices are end-of-life, and which signing keys or certificates are embedded.

The first step may be crypto agility: allowing devices to understand multiple algorithms and metadata before enforcing PQC signatures.

Model inventoryWhat is deployed.
VerifierAccepted algorithms.
Update pathCan it change?
ReplacementWhen update is impossible.

Concrete example

A smart meter bootloader that only verifies ECDSA may need a bootloader update before any PQC-signed firmware can be accepted.

Common pitfall

Do not design a signing policy that field devices cannot verify.

Takeaway

Device verifier capability determines migration reality.
