Developer Implementation Engineering

The gap between "the library supports ML-KEM" and "we shipped PQC safely" is full of practical engineering: sizes, serialization, randomness, side channels, and hardware dependencies. Here is what actually bites.

  • Use a PQC library correctly and recognize what "supported" leaves out.
  • Handle the new sizes in schemas, buffers, and wire formats.
  • Respect randomness, side-channel, and HSM/KMS constraints.

Try it: a KEM is three calls

The API is small; the discipline around it is the work. Here is ML-KEM key establishment with the liboqs Python binding.

# pip install liboqs-python  (bindings for the Open Quantum Safe liboqs)
import oqs

kem = "ML-KEM-768"
# Recipient generates a keypair
with oqs.KeyEncapsulation(kem) as recipient:
    public_key = recipient.generate_keypair()        # 1184 bytes

    # Sender encapsulates to the public key
    with oqs.KeyEncapsulation(kem) as sender:
        ciphertext, secret_sender = sender.encap_secret(public_key)  # ct 1088 B

    # Recipient decapsulates to recover the SAME secret
    secret_recipient = recipient.decap_secret(ciphertext)

assert secret_sender == secret_recipient   # now feed this into AES-256-GCM
The lesson

The shared secret is 32 bytes — you still hand it to a KDF and symmetric cipher. The KEM only replaces the "agree on a secret" step. Everything you learned about AES-GCM still applies.

1

Library readiness is more than "algorithm available"

"We have an ML-KEM function" is the start of a checklist, not the end.

  • Validation: is the implementation FIPS-validated (or on a path to it) where you need that? A correct algorithm is not automatically an approved module.
  • Test vectors: does it pass the official NIST/ACVP known-answer tests for the exact parameter set?
  • Constant-time: is decapsulation and signing implemented without secret-dependent branches/memory access?
  • Maintenance: who patches it, how fast, and is it pinned to a reviewed version?
  • API stability: pre-standard "Kyber" APIs differ from final "ML-KEM"; do not ship a draft.

"Supported" is a spectrum, and the gap between the ends is where projects get burned. At the shallow end, a library merely compiles an ML-KEM function. At the deep end, it is a FIPS-validated module, passes the official ACVP test vectors for your exact parameter set, runs decapsulation and signing in constant time, is actively maintained with a fast patch cadence, and exposes the final standardized names and encodings. Write these out as acceptance criteria and check each one — do not let "it has the function" stand in for "it is production-ready."

Concrete example

Two teams both "support ML-KEM." Team A pulled a research branch that still uses round-3 Kyber constants; Team B uses a maintained, FIPS-track build of FIPS 203. Their handshakes will not interoperate, and Team A's build may never be approvable for regulated use. Same words on the slide, completely different reality — which is why the algorithm identifier and version belong in your inventory.

Common pitfall

Shipping a pre-standardization implementation (round-3 Kyber) and assuming it interoperates with FIPS 203 ML-KEM. Parameters and encodings changed; they do not match.

2

Metadata, serialization, and storage

The number-one integration bug is a field that is too small.

Classical assumptions are baked into schemas everywhere: a 256-byte column "big enough for any signature," a 2 KB handshake buffer, a fixed firmware header slot. PQC blows past these. Audit every place a key, ciphertext, or signature is stored or transmitted.

Artifact (example)ClassicalPQCWhat to check
SignatureECDSA ~72 BML-DSA-65 ~3,309 BDB columns, headers, token limits
KEM public keyX25519 32 BML-KEM-768 1,184 BHandshake buffers, caches
KEM ciphertextX25519 share 32 BML-KEM-768 1,088 BMessage/packet size limits
Signing keyP-256 32 BSLH-DSA large state / seedsHSM object sizes, key stores
Concrete example

A JWT-style token that embedded an ECDSA signature now must carry a 3 KB ML-DSA signature. Header size limits on proxies, cookies, and some HTTP servers can reject it — a failure that looks like a networking bug, not a crypto bug.

Practice

Store an explicit algorithm identifier next to every key and signature. Agility is impossible if you cannot tell which algorithm produced a stored artifact.

Here is why size is the sneaky killer: cryptographic artifacts do not live in one place. A single signature might be generated in an HSM, serialized into a certificate, embedded in a token, cached in a database, logged for audit, replicated to a backup, and re-parsed by a dozen services — each with its own idea of "how big could this possibly get." Any one of those with a hard-coded limit becomes a landmine that only detonates in production, often intermittently, and usually looks like an unrelated networking or database error. The fix is boring but reliable: treat "audit every size assumption" as a first-class migration task, not an afterthought.

Everyday analogy

Imagine your postal system was built for postcards and someone suddenly starts mailing thick parcels. The letters still say the same thing, but now they jam the sorting machines, do not fit the slots, and overflow the mailboxes. PQC keys and signatures are those parcels: the meaning is identical, but every pipe they pass through has to be widened to carry them.

3

Randomness and side-channel safety

PQC schemes are as sensitive to bad randomness and timing leaks as classical ones — sometimes more.

Key generation and encapsulation need a strong CSPRNG seeded from the OS. On constrained/embedded devices, weak entropy at first boot is a real failure mode. Separately, secret-dependent timing or memory-access patterns can leak private keys; that is why decapsulation and signing must be constant-time.

Embedded and IoT devices deserve special caution on two fronts. First, entropy at first boot: a device that generates keys immediately after manufacture, before it has gathered enough randomness, can produce weak or even duplicated keys across a whole production run — a failure that has bitten real fleets. Seed from a hardware RNG and, if needed, defer key generation until entropy is sufficient. Second, physical side channels: an attacker with the device in hand can measure power draw or electromagnetic emissions during decapsulation or signing, or inject faults to induce revealing errors. These are not theoretical for high-value or widely distributed devices, and they require hardened, evaluated implementations — not just constant-time software.

What a side channel actually is

A side channel is information that leaks not from what a program computes but from how it computes — how long it takes, how much power it draws, what sounds or emissions it makes. Imagine guessing a safe's combination by listening to the tumblers, or learning someone's PIN from how long their finger hovers over each key. The secret was never "revealed," yet it leaked anyway. Constant-time code closes the timing version of this leak by making every code path take the same time regardless of the secret; hardened hardware closes the power and electromagnetic versions.

Common pitfall

Rolling your own reference implementation "for speed." Naive lattice code often leaks through timing. Use reviewed, constant-time libraries and, on devices, evaluate for physical side channels (power/EM).

The uncomfortable lesson from decades of classical crypto is that the algorithm is almost never where real systems get broken. AES and RSA were rarely defeated by attacking the math; they were defeated by leaked keys, bad randomness, timing side channels, and implementation bugs. PQC inherits this reality and, because its operations are newer and more complex, arguably widens the attack surface for implementation mistakes. That is why this entire module is about handling, sizing, randomness, and side channels rather than lattice theory: getting the plumbing right is what actually keeps you safe.

Go deeper: decapsulation failures must not leak

ML-KEM uses an "implicit rejection" design: if a ciphertext is malformed, decapsulation still returns a pseudo-random secret rather than an obvious error, so an attacker cannot tell valid from invalid by observing behavior. If your integration adds an early error return or a distinguishable log/timing on failure, you can reintroduce a chosen-ciphertext oracle the algorithm was designed to prevent. Do not "helpfully" branch on decapsulation outcome.

4

HSM, KMS, TPM, secure element, and cloud dependencies

Your private keys often live in hardware or a managed service — and those move on their own timeline.

If an HSM, cloud KMS, TPM, or secure element does not yet support a PQC algorithm, you cannot generate or use PQC keys there no matter how ready your code is. This is frequently the true bottleneck. Track firmware/service roadmaps as first-class dependencies, and design so the key-storage backend is swappable.

App codeReady for ML-KEM/ML-DSA.
→
KMS/HSMMust support the algorithm + sizes.
→
Device SE/TPMMay need firmware or hardware refresh.
→
RolloutGated by the slowest dependency.
Takeaway

Safe rollout: feature-flag PQC, run hybrid first, canary a small population, watch handshake/verify failure metrics, and keep a fast rollback. Never big-bang a crypto change.

5

Designing for crypto-agility in code

The goal is not "use ML-KEM." It is "be able to change algorithms later without a rewrite."

Most legacy code hard-wires the algorithm: rsa_sign(...) or ecdh(...) scattered across the codebase, with key sizes and encodings assumed inline. That is exactly what made this migration a multi-year project. The antidote is an abstraction layer: application code asks for a capability ("sign this", "establish a key"), and a small provider layer picks the concrete algorithm from configuration and policy.

# Application asks for a capability; policy decides the algorithm.
signer = crypto.get_signer(policy="default-2026")   # -> ML-DSA-65 today
sig    = signer.sign(message)
blob   = {"alg": signer.alg_id, "sig": sig}    # store the alg ID with it

# Next year, switch to a backup family with ONE policy change:
#   policy "default-2027" -> SLH-DSA, no application code edits.
Common pitfall

Treating agility as "we'll refactor when we need to." You need it before the emergency. Build the abstraction now, while there is no time pressure, so a future algorithm break is a config change instead of a fire drill.

6

Performance and sizing budgets

PQC is fast enough for almost everything — but the bytes, not the CPU, are what you must budget.

A frequent worry is that PQC will be too slow. In practice, lattice operations (ML-KEM, ML-DSA) are competitive with or faster than RSA on modern CPUs; the real cost is size. The table below gives representative relative figures to build intuition — measure your own stack, but expect the pattern to hold.

ConcernClassical baselinePQC (typical Level 3)Practical impact
Key-exchange CPUX25519: very fastML-KEM-768: comparable, often faster than RSARarely the bottleneck
Signature verify CPUECDSA: fastML-DSA-65: fast; SLH-DSA: slowerWatch SLH-DSA at high verify volume
Handshake bytes~1–2 KB total+~1–2 KB (KEM) to +many KB (PQC certs)The real budget item — extra round trips, packet loss
Storage per key/sigtens–hundreds of bytes~1–8 KBDB growth, cache pressure, token limits
Concrete example

On a busy TLS terminator, moving to hybrid ML-KEM typically costs negligible extra CPU but adds ~1 KB per handshake. Add PQC certificates and a chain can grow by several KB, which can push the handshake across TCP's initial congestion window and add a round trip on lossy mobile links. Budget bytes and round trips, not gigahertz.

Practice

Add a "handshake byte budget" and a "max artifact size" to your test suite, and alert if a PQC change blows past it. Sizing regressions are the failures that reach production disguised as flaky networking.

Hands-on labs

Decide first, then reveal.

Lab 1 · The 256-byte column

Auth tokens store a signature in a VARCHAR(512) field; you switch to ML-DSA-65.

What happens, and how do you find every instance?

Recommended answer The ~3.3 KB signature is truncated or rejected. Grep schemas/protobuf/IDL for fixed sizes, audit token/cookie/header limits, and add size tests to CI. Store an algorithm ID alongside each signature.

Lab 2 · "Let's optimize decapsulation"

A dev adds an early return error when a ciphertext looks malformed, to save time.

Why is this dangerous?

Recommended answer It can create a timing/behavior oracle that distinguishes valid from invalid ciphertexts, undermining ML-KEM's implicit-rejection design. Keep decapsulation constant-time and non-branching on the secret result.

Lab 3 · The HSM can't

Your code is PQC-ready but the HSM firmware has no ML-DSA.

What is the real status and next step?

Recommended answer You are blocked on a hardware dependency, not code. Record it as the critical-path item, get the vendor firmware roadmap, and keep the key backend swappable so you can cut over when support ships.

Lab 4 · Retrofit agility

A codebase has ecdsa_sign() and hard-coded key sizes called from 40 different files.

What is the highest-leverage first change — before touching any algorithm?

Recommended answer Introduce a crypto provider abstraction: replace the 40 direct calls with a capability interface (signer.sign()) that selects the algorithm from policy, and store an algorithm ID with every artifact. Once that layer exists, adopting ML-DSA — or a future backup — becomes a config change instead of a 40-file refactor under time pressure.

Lab 5 · "PQC is too slow"

A stakeholder resists PQC citing CPU cost.

Is CPU the right concern, and what should you actually budget?

Recommended answer Usually not. Lattice KEM/signature CPU is competitive with (often faster than) RSA. The real budget item is size: larger handshakes and artifacts that can add round trips and stress buffers, tokens, and storage. Add byte-size budgets to tests; do not over-index on gigahertz.
