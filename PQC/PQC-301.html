Developer Implementation Engineering

The gap between "the library supports ML-KEM" and "we shipped PQC safely" is full of practical engineering: sizes, serialization, randomness, side channels, and hardware dependencies. Here is what actually bites.

  • Use a PQC library correctly and recognize what "supported" leaves out.
  • Handle the new sizes in schemas, buffers, and wire formats.
  • Respect randomness, side-channel, and HSM/KMS constraints.

Try it: a KEM is three calls

The API is small; the discipline around it is the work. Here is ML-KEM key establishment with the liboqs Python binding.

# pip install liboqs-python  (bindings for the Open Quantum Safe liboqs)
import oqs

kem = "ML-KEM-768"
# Recipient generates a keypair
with oqs.KeyEncapsulation(kem) as recipient:
    public_key = recipient.generate_keypair()        # 1184 bytes

    # Sender encapsulates to the public key
    with oqs.KeyEncapsulation(kem) as sender:
        ciphertext, secret_sender = sender.encap_secret(public_key)  # ct 1088 B

    # Recipient decapsulates to recover the SAME secret
    secret_recipient = recipient.decap_secret(ciphertext)

assert secret_sender == secret_recipient   # now feed this into AES-256-GCM
The lesson

The shared secret is 32 bytes — you still hand it to a KDF and symmetric cipher. The KEM only replaces the "agree on a secret" step. Everything you learned about AES-GCM still applies.

1

Library readiness is more than "algorithm available"

"We have an ML-KEM function" is the start of a checklist, not the end.

  • Validation: is the implementation FIPS-validated (or on a path to it) where you need that? A correct algorithm is not automatically an approved module.
  • Test vectors: does it pass the official NIST/ACVP known-answer tests for the exact parameter set?
  • Constant-time: is decapsulation and signing implemented without secret-dependent branches/memory access?
  • Maintenance: who patches it, how fast, and is it pinned to a reviewed version?
  • API stability: pre-standard "Kyber" APIs differ from final "ML-KEM"; do not ship a draft.
Common pitfall

Shipping a pre-standardization implementation (round-3 Kyber) and assuming it interoperates with FIPS 203 ML-KEM. Parameters and encodings changed; they do not match.

2

Metadata, serialization, and storage

The number-one integration bug is a field that is too small.

Classical assumptions are baked into schemas everywhere: a 256-byte column "big enough for any signature," a 2 KB handshake buffer, a fixed firmware header slot. PQC blows past these. Audit every place a key, ciphertext, or signature is stored or transmitted.

Concrete example

A JWT-style token that embedded an ECDSA signature now must carry a 3 KB ML-DSA signature. Header size limits on proxies, cookies, and some HTTP servers can reject it — a failure that looks like a networking bug, not a crypto bug.

Practice

Store an explicit algorithm identifier next to every key and signature. Agility is impossible if you cannot tell which algorithm produced a stored artifact.

3

Randomness and side-channel safety

PQC schemes are as sensitive to bad randomness and timing leaks as classical ones — sometimes more.

Key generation and encapsulation need a strong CSPRNG seeded from the OS. On constrained/embedded devices, weak entropy at first boot is a real failure mode. Separately, secret-dependent timing or memory-access patterns can leak private keys; that is why decapsulation and signing must be constant-time.

Common pitfall

Rolling your own reference implementation "for speed." Naive lattice code often leaks through timing. Use reviewed, constant-time libraries and, on devices, evaluate for physical side channels (power/EM).

Go deeper: decapsulation failures must not leak

ML-KEM uses an "implicit rejection" design: if a ciphertext is malformed, decapsulation still returns a pseudo-random secret rather than an obvious error, so an attacker cannot tell valid from invalid by observing behavior. If your integration adds an early error return or a distinguishable log/timing on failure, you can reintroduce a chosen-ciphertext oracle the algorithm was designed to prevent. Do not "helpfully" branch on decapsulation outcome.

4

HSM, KMS, TPM, secure element, and cloud dependencies

Your private keys often live in hardware or a managed service — and those move on their own timeline.

If an HSM, cloud KMS, TPM, or secure element does not yet support a PQC algorithm, you cannot generate or use PQC keys there no matter how ready your code is. This is frequently the true bottleneck. Track firmware/service roadmaps as first-class dependencies, and design so the key-storage backend is swappable.

KMS/HSMMust support the algorithm + sizes.

Takeaway

Safe rollout: feature-flag PQC, run hybrid first, canary a small population, watch handshake/verify failure metrics, and keep a fast rollback. Never big-bang a crypto change.

Hands-on labs

Decide first, then reveal.

Lab 1 · The 256-byte column

Auth tokens store a signature in a VARCHAR(512) field; you switch to ML-DSA-65.

What happens, and how do you find every instance?

Recommended answer The ~3.3 KB signature is truncated or rejected. Grep schemas/protobuf/IDL for fixed sizes, audit token/cookie/header limits, and add size tests to CI. Store an algorithm ID alongside each signature.

Lab 2 · "Let's optimize decapsulation"

A dev adds an early return error when a ciphertext looks malformed, to save time.

Why is this dangerous?

Recommended answer It can create a timing/behavior oracle that distinguishes valid from invalid ciphertexts, undermining ML-KEM's implicit-rejection design. Keep decapsulation constant-time and non-branching on the secret result.

Lab 3 · The HSM can't

Your code is PQC-ready but the HSM firmware has no ML-DSA.

What is the real status and next step?

Recommended answer You are blocked on a hardware dependency, not code. Record it as the critical-path item, get the vendor firmware roadmap, and keep the key backend swappable so you can cut over when support ships.
